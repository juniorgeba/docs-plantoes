Plantões API

API REST para gerenciamento de plantões médicos. Suporta autenticação via Bearer Token (Laravel Sanctum), controle de vagas, check-in/check-out com foto e GPS, e fluxo de trocas entre médicos.

Versão v1
Atualizado em 11/05/2026
Protocolo HTTPS
Formato JSON
Auth Bearer + API Key
Novidades — 11/05/2026
Campos financeiros no plantão médico com snapshot histórico.
💰
4 novas colunas na tabela plantoes
valor_base (DECIMAL 10,2, nullable) — valor total calculado no check-out.
valor_hora (DECIMAL 10,2, nullable) — snapshot do R$/hora vigente no momento do check-out.
valor_complemento (DECIMAL 10,2, nullable) — snapshot do complemento vigente no check-out.
valor_por_servico (DECIMAL 10,2, nullable) — snapshot do valor por serviço vigente no check-out.
Execute: php artisan migrate
👤
3 novas colunas na tabela dados_profissionais
valor_hora, valor_complemento e valor_por_servico (DECIMAL 10,2, nullable) — configurados pelo admin no cadastro do médico. São a fonte dos valores, mas não devem ser usados para exibir histórico de pagamento — use os campos do próprio plantão para isso.
📸
Padrão snapshot — por que os valores ficam no plantão?
Os campos valor_hora, valor_complemento e valor_por_servico são gravados diretamente na tabela plantoes no momento do check-out — como um e-commerce que registra o preço no pedido, não referencia o preço atual do produto. Isso garante que alterações futuras no cadastro financeiro do médico nunca alterem o histórico de plantões já realizados. O frontend deve sempre exibir plantao.valor_hora (e não medico.dados_profissionais.valor_hora) ao mostrar o breakdown de um plantão concluído.
⚙
Configuração do arredondamento via .env
CALCULO_ARREDONDAMENTO_MINUTOS=30 — granularidade do arredondamento (ex: 15, 30, 60 minutos).
CALCULO_ARREDONDAMENTO_TIPO=half — regra: half (arredonda quando resto ≥ 50% da granularidade), up (sempre sobe), down (sempre trunca).
O cálculo é 100% server-side — o frontend não precisa calcular nada, apenas exibir o valor_base retornado pela API.
📡
Impacto na API
Os endpoints GET /api/v1/plantoes e GET /api/v1/plantoes/{id} agora retornam 4 novos campos no objeto plantão: valor_base, valor_hora, valor_complemento e valor_por_servico. Todos são float|null — serão null enquanto o plantão não tiver check-out ou se o médico não tiver tarifa configurada.
O endpoint POST /plantoes/{id}/check-out também passa a retornar valor_base na resposta de sucesso. Nenhum campo novo é necessário no body da requisição — o cálculo é automático.
Novidades — 08/05/2026
Resumo das alterações desta versão em relação à documentação anterior.
📎
Anexo de documento no check-out (módulo web)
O check-out via interface web agora aceita um arquivo opcional de até 5MB como comprovante (lista de presença, atestado, etc.). Tipos permitidos: PDF, JPG, PNG, DOC, DOCX, XLS, XLSX, ODT, ODS, TXT. O arquivo é armazenado em storage/app/private/plantoes/anexos/ — não fica exposto publicamente. Download autenticado via rota web GET /meus-plantoes/{id}/anexo.
ℹ
Novo campo tem_anexo_checkout na API
O objeto plantão retornado pelos endpoints GET /api/v1/plantoes e GET /api/v1/plantoes/{id} agora inclui o campo booleano tem_anexo_checkout. Use-o no app React para exibir um indicador ou botão de download quando o médico anexou um documento ao fazer check-out.
🗄
Nova coluna na tabela plantoes
Coluna check_out_anexo (VARCHAR 500, nullable) — armazena o caminho relativo do arquivo dentro do storage. Execute a migration: php artisan migrate.
Versão anterior — 07/05/2026
⚠
Validação de conflito de horário
Os endpoints POST /vagas/{id}/assumir, POST /plantoes/{id}/solicitar-troca e POST /trocas/{id}/aceitar agora verificam sobreposição de horário antes de concluir a operação. Se o médico já tiver plantão ativo no mesmo dia com horário conflitante, a API retorna HTTP 422 com a mensagem de conflito. Ver detalhes nas seções de cada endpoint.
ℹ
Objeto conflito no assumir
O endpoint POST /vagas/{id}/assumir agora retorna, junto com a mensagem de erro 422, um objeto conflito com os detalhes do plantão que está causando o bloqueio (id, data, horários, hospital e especialidade). Use esse objeto para exibir informação precisa ao médico no app.
👥
Status inativos que não bloqueiam
A verificação de sobreposição ignora plantões com status cancelado, troca_com_colega, nao_realizada e nao_compareceu. Apenas plantões ativos bloqueiam a atribuição.
Versão anterior — 01/05/2026
📷
Check-in e Check-out com Foto + GPS
Os endpoints POST /plantoes/{id}/check-in e POST /plantoes/{id}/check-out aceitam dois campos opcionais: foto (base64 da câmera) e coordenadas (GPS em formato "lat,lng").
🗃
Novos campos na tabela plantoes
check_in_foto e check_out_foto como LONGTEXT, e check_in_coordenadas e check_out_coordenadas como VARCHAR(50). Importante: use LONGTEXT (não TEXT) para fotos — TEXT tem limite 65KB e causa erro 1406.
🌎
Rota web de download de foto: GET /meus-plantoes/{id}/foto/{tipo} — download da foto de presença. Tipo aceito: checkin ou checkout. Autenticação por sessão web (não Bearer).
Autenticação
A API suporta dois métodos de autenticação conforme o endpoint.
Base URL https://plantoes-app.zml1g1.easypanel.host/api/v1
⚠
Sempre inclua o header Accept: application/json em todas as requisições. Sem ele, o Laravel pode retornar HTML em erros de validação.
# Rotas protegidas por Bearer Token (Sanctum)
Authorization: Bearer {seu_token}
Accept: application/json
Content-Type: application/json

# Rota protegida por API Key (sistema externo)
X-API-Key: {sua_chave}
Accept: application/json
🔑
As rotas GET /vagas/quantidade-total, GET /vagas/panorama-disponiveis e GET /vagas/panorama-preenchidas usam X-API-Key no header em vez de Bearer Token. A chave é configurada via variável de ambiente API_KEY no servidor.
ℹ
O Bearer Token expira em 30 dias após a emissão. Ao fazer login novamente com o mesmo device_name, o token anterior é revogado automaticamente.
Códigos de Status
A API usa os códigos HTTP padrão para indicar sucesso ou falha.
200OK — Sucesso
201Created — Recurso criado
401Unauthorized — Token inválido
403Forbidden — Sem permissão
404Not Found — Recurso não existe
422Unprocessable — Validação falhou
// Exemplo de erro de validação (422)

  }

POST /api/v1/vagas/{id}/assumir Assumir uma vaga disponível 🔒 Bearer ▶

Não requer body. O médico deve possuir a especialidade da vaga. Apenas usuários com is_medico: true podem assumir.

⚠
Verificação de sobreposição de horário — antes de criar o plantão, o sistema verifica se o médico já tem plantão ativo no mesmo dia com horário conflitante. Se houver, retorna 422 com o objeto conflito descrevendo o plantão bloqueador.
// Sucesso 201
{ "message": "Vaga assumida com sucesso!" }

// 403 — usuário não é médico
{ "message": "Apenas médicos podem assumir vagas." }

// 403 — não possui a especialidade
{ "message": "Você não possui a especialidade requerida." }

// 422 — vaga não está disponível
{ "message": "Vaga não está disponível." }

// 422 — vaga já foi assumida
{ "message": "Esta vaga já foi assumida." }

// 422 — conflito de horário com outro plantão do médico
{
  "message": "Conflito de horário: Dr. João Silva já possui plantão neste dia (15/05/2026) das 07:00 às 19:00 em Hospital São Lucas (Clínica Médica). Verifique a escala antes de prosseguir.",
  "conflito": {
    "plantao_id": 88,
    "data": "2026-05-15",
    "hora_inicio": "07:00",
    "hora_fim": "19:00",
    "hospital": "Hospital São Lucas",
    "especialidade": "Clínica Médica"
  }
}
CondiçãoDetalhe
is_medicoO usuário autenticado deve ter is_medico: true, caso contrário retorna 403.
EspecialidadeO médico deve possuir a especialidade da vaga na tabela especialidade_usuario.
Status da vagaA vaga deve estar com status disponivel.
SobreposiçãoO sistema verifica se o médico tem algum plantão ativo (status diferente de cancelado, troca_com_colega, nao_realizada, nao_compareceu) na mesma data com horário que se intersecte com o da vaga. Plantões que cruzam meia-noite são tratados corretamente.
AtomicidadeA criação do plantão e a atualização do status da vaga para preenchida ocorrem dentro de uma única DB::transaction.

POST /api/v1/plantoes/{id}/check-in Realizar check-in no plantão 🔒 Bearer ▶
ℹ

DetalheInformação
Formato da fotoAceita Data URI completo (data:image/jpeg;base64,…) ou base64 puro. O servidor detecta automaticamente e armazena como recebido.
Tamanho máximoUma foto JPEG 1280×960 a 85% de qualidade resulta em ~100–200KB de base64. A coluna é LONGTEXT (limite ~4GB). Considere comprimir no cliente para economizar banda.

POST /api/v1/plantoes/{id}/check-out Finalizar plantão com check-out 🔒 Bearer ▶

Plantão deve estar com status em_execucao. Calcula divergência automaticamente comparando com a duração prevista da vaga.

💰
Cálculo financeiro automático: se o médico tiver valor_hora configurado, o servidor calcula automaticamente o valor_base e grava também um snapshot de valor_hora, valor_complemento e valor_por_servico diretamente no registro do plantão. Nenhum campo extra é necessário no body — o cálculo é 100% server-side. O valor_base será null se o médico não tiver tarifa configurada.
📎
Endpoint aceita dois formatos de envio: application/json (sem anexo) ou multipart/form-data (quando há arquivo). Use sempre multipart/form-data no app React para simplificar — campos de texto funcionam normalmente em ambos os formatos.

coordenadasstringopc.GPS no formato "lat,lng". Ex: "-23.550520,-46.633308".
anexofileopc.Documento comprobatório (lista de presença, atestado, etc.). Máx. 5MB. Tipos aceitos: pdf, jpg, jpeg, png, doc, docx, xls, xlsx, odt, ods, txt. Requer Content-Type: multipart/form-data.

ℹ

POST /api/v1/trocas/{id}/aceitar Aceitar solicitação de troca 🔒 Bearer ▶

Não requer body. Cria automaticamente um novo plantão para o substituto e atualiza o status do plantão original para troca_com_colega.

⚠
Verificação de sobreposição no momento de aceitar — ao aceitar, o sistema re-verifica se o médico (o substituto que está aceitando) ainda não tem conflito de horário naquela data. Se entre a solicitação e o aceite o médico tiver assumido outro plantão conflitante, o aceite será bloqueado com 422. A verificação ignora o plantão original da troca para não gerar falso conflito consigo mesmo.

{ "message": "No query results for model [SolicitacaoTroca]." }
CondiçãoDetalhe
Dono da solicitaçãoA query filtra por substituto_id = usuario_id, portanto apenas o médico que recebeu a solicitação pode aceitá-la.
Status da solicitaçãoApenas solicitações com status pendente podem ser aceitas.
SobreposiçãoRe-verifica conflito de horário no momento do aceite. O plantão original da troca é excluído da checagem para evitar falso positivo.
EfeitoPlantão original → troca_com_colega. Novo plantão criado para o substituto com status prevista. Solicitação → aceita. Tudo em DB::transaction.
