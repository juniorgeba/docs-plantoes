Plantões API

API REST para gerenciamento de plantões médicos. Suporta autenticação via Bearer Token (Laravel Sanctum), controle de vagas, check-in/check-out e fluxo de trocas entre médicos.

Versão v1
Protocolo HTTPS
Formato JSON
Auth Bearer Token
Autenticação
Todas as rotas (exceto login) requerem um Bearer Token no header Authorization.
Base URL https://plantoes-app.zml1g1.easypanel.host/api/v1
⚠
Sempre inclua o header Accept: application/json em todas as requisições. Sem ele, o Laravel pode retornar HTML em erros de validação.
# Headers obrigatórios para rotas protegidas
Authorization: Bearer {seu_token}
Accept: application/json
Content-Type: application/json
ℹ
O token expira em 30 dias após a emissão. Ao fazer login novamente com o mesmo device_name, o token anterior é revogado automaticamente.
Códigos de Status
A API usa os códigos HTTP padrão para indicar sucesso ou falha.
200OK — Sucesso
201Created — Recurso criado
401Unauthorized — Token inválido
403Forbidden — Sem permissão
404Not Found — Recurso não existe
422Unprocessable — Validação falhou
// Exemplo de erro de validação (422)
{
  "message": "The email field is required.",
  "errors": {
    "email": ["The email field is required."]
  }
}

POST /api/v1/vagas/{id}/assumir Assumir uma vaga disponível 🔒 Bearer ▶

Não requer body. O médico deve possuir a especialidade da vaga. Apenas usuários com is_medico: true podem assumir.

// Sucesso 201

// Erros possíveis
// 403 — Não é médico ou não tem a especialidade
// 422 — Vaga indisponível ou já assumida

GET /api/v1/plantoes/{id} Detalhar um plantão específico 🔒 Bearer ▶

Retorna o mesmo objeto do item da listagem. O médico só pode acessar seus próprios plantões (404 se de outro usuário).

POST /api/v1/plantoes/{id}/check-in Realizar check-in no plantão 🔒 Bearer ▶
ℹ
Check-in disponível apenas entre 30 minutos antes e 2 horas após o início do plantão. Status deve ser prevista.
// Sucesso 200
{
  "message": "Check-in realizado com sucesso!",
  "check_in_at": "07:03"
}

// Erro 422 — fora da janela de tempo
{ "message": "Check-in disponível apenas a partir de 30 minutos antes do início do plantão." }
POST /api/v1/plantoes/{id}/check-out Finalizar plantão com check-out 🔒 Bearer ▶

Plantão deve estar com status em_execucao. Calcula divergência automaticamente comparando com a duração prevista da vaga.

// Sucesso 200
{
  "message": "Check-out realizado! Plantão concluído.",
  "duracao_realizada": "12h5min",
  "diferenca": "+5min",
  "divergencia": true
}
POST /api/v1/plantoes/{id}/solicitar-troca Solicitar troca de plantão com outro médico 🔒 Bearer ▶
CampoTipoReq.Descrição
substituto_idinteger✱ obr.ID do médico substituto (deve existir e ter a especialidade)
 
{ "substituto_id": 55 }
// Sucesso 200
{ "message": "Solicitação de troca enviada!" }

// 422 — status inválido
{ "message": "Só é possível trocar plantões previstos." }

// 422 — especialidade incompatível
{ "message": "O médico não possui a especialidade requerida." }

POST /api/v1/trocas/{id}/aceitar Aceitar solicitação de troca 🔒 Bearer ▶

Não requer body. Cria automaticamente um novo plantão para o substituto e atualiza o status do plantão original para troca_com_colega.

// Sucesso 200
{ "message": "Troca aceita! O plantão está confirmado para você." }
POST /api/v1/trocas/{id}/recusar Recusar solicitação de troca 🔒 Bearer ▶

Reverte o plantão original para status prevista.

// Sucesso 200
{ "message": "Troca recusada." }
